## What this project is

`multi-api-release` is a **NestJS CLI tool** (not an HTTP server) that orchestrates production releases of microservices in the `multi-develop` GitHub org. It tags Git repos, records releases in a PostgreSQL database, and exposes both an interactive TUI menu and subcommands via Commander.

## Commands

```bash
# Build
pnpm build                       # compiles to dist/

# Run CLI (dev — ts-node, no build needed)
pnpm cli:dev                     # interactive menu
pnpm cli:dev db migrate          # run pending SQL migrations
pnpm cli:dev db seed             # seed services from repository-list.txt
pnpm cli:dev service list
pnpm cli:dev service info <name>
pnpm cli:dev release create <service> [--services s1,s2] [-y] [--dry-run] [-n notes]
pnpm cli:dev release list        [-s <service>] [-l <limit>]
pnpm cli:dev release status <id|latest>

# Run CLI (production — requires pnpm build first)
pnpm cli db migrate
pnpm cli release create api-users -y

# Lint / format
pnpm lint
pnpm format
```

No test scripts are configured yet (`jest` is not installed).

## Environment

Copy `.env.example` to `.env` and fill in:

| Variable | Description |
|---|---|
| `DB_HOST/PORT/USERNAME/PASSWORD/DATABASE` | PostgreSQL connection (DB: `multiapp-release`, schema: `releases`) |
| `GH_MULTI_TOKEN` | GitHub PAT — load from `source ~/.claude/profile.sh` |
| `GH_ORG` | GitHub org (default: `multi-develop`) |
| `REPO_LIST_PATH` | Path to `repository-list.txt` (default: sibling project `multi-service-k8s`) |

## Architecture

### Startup flow (`src/main.ts`)

- **No args** → interactive mode via `InactiveCommand` (Inquirer menu)
- **`-h/--help/-V`** → prints help without bootstrapping NestJS (fast path)
- **Any other args** → full NestJS context + `setupCli()` dispatches Commander subcommands

### Module graph

```
AppModule
├── ConfigModule (global)          → src/config/app.config.ts
├── DatabaseModule                 → TypeORM + PostgreSQL, schema: releases, synchronize: false
├── ServicesModule                 → ServicesService + ServiceRepository
├── ReleasesModule                 → ReleasesService + ReleaseRepository + ReleaseServiceRepository
├── GithubModule                   → GithubService (Octokit) — reads GH_MULTI_TOKEN at onModuleInit
└── CliModule                      → all Command classes (DI-injected)
```

### Key design decisions

- **No HTTP transport** — `NestFactory.createApplicationContext()` is used, not `NestFactory.create()`.
- **Repository Pattern** — each entity has its own repository class (`src/*/repositories/`) wrapping TypeORM `DataSource`/`Repository`.
- **Manual SQL migrations** — `DbMigrateCommand` reads `.sql` files from `shared/sql/` in alphabetical order and tracks applied files in `releases.schema_migrations`. TypeORM `synchronize: false`.
- **Version format** — `vYYYY.MMDD.HHmm` (e.g. `v2026.0504.1430`) generated by `src/common/utils/version.util.ts`.
- **Tag naming** — Git tags are the version string; image tags use the pattern `prd.<service-name>.release`.

### SQL migrations location

`shared/sql/` — four files applied in order:
1. `001-create-schema.sql`
2. `002-create-types.sql`
3. `003-create-tables.sql`
4. `004-create-indexes.sql`

### Data model (`releases` schema)

- `service` — catalog of microservices (name, repository, tier, isActive)
- `release` — a release event (version, status enum, notes, createdBy)
- `release_service` — join table linking a release to specific services with gitSha and imageTag

### GitHub integration

`GithubService` wraps `@octokit/rest` to:
1. `getBranchSha(repo, branch)` — resolve HEAD SHA of `master`
2. `createTag(repo, tagName, sha)` — create a lightweight Git tag ref

### Output utilities (`src/common/utils/output.util.ts`)

All CLI output goes through `success`, `error`, `info`, `warn` (chalk-colored), `createSpinner` (ora), and `createTable` (cli-table3). Use these instead of raw `console.log`.

## Adding a new command

1. Create `src/cli/commands/<name>.command.ts` implementing `ICommand` (see `src/cli/interfaces/command.interface.ts`).
2. Declare it as `@Injectable()` and add it to `CliModule` providers.
3. Register the Commander subcommand in `src/cli/cli.setup.ts`.
4. Add it to the interactive menu in `src/cli/commands/interactive.command.ts` if needed.
